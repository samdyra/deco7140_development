Accessibility Assessment

1. Alternative Text for Non-text Content (WCAG 1.1.1 - Level A)

User Need Identified:

From Lily (Visually Impaired person): "small and I cannot make it bigger easily. Also, many posts have photos with text inside, and I cannot read those at all."

From Client Brief (Value 2): "Provide clear, accessible information and resources to users."

Success Criteria:

"All non-text content that is presented to the user has a text alternative that serves the equivalent purpose"

Implementation on My Website:

All the content created such as posts or recipes, will include alternative texts to explain the dish. Same things with other contents, such as event photos, or user activities in communities etc.

2. Contrast (Minimum) (WCAG 1.4.3 - Level AA)

User Need Identified:

From Lily: "many websites use gray text on white, or white text on light colors - I cannot read this at all!"

From Client Brief (Value 4): "Ensure ethical, inclusive, and culturally sensitive design principles are upheld."

Success Criteria:

"The visual presentation of text and images of text has a contrast ratio of at least 4.5:1, except for Large-scale text and images of large-scale text have a contrast ratio of at least 3:1; "

Implementation on My Website:

I will create a proper color scheme that has a good contrast ratio between the colors. Furtheremore, I will make sure the usage of this color scheme is appropriate too, by creating UI components that follow this color scheme. Therefore the rest of the app will all have good contrast ratio, as long it use the UI components.

3. Resize Text (WCAG 1.4.4 - Level AA)

User Need Identified:

From Lily: "I need big letters, or at least option to make them bigger. Many websites have tiny text and when I try to zoom, everything becomes... messy? Words go outside the page, buttons disappear. This is very frustrating."

From Client Brief (Value 4): "Accessibility must be prioritised to ensure seamless interaction for all users, including those relying on assistive technologies."

Success Criteria:

"Except for captions and images of text, text can be resized without assistive technology up to 200 percent without loss of content or functionality."

Implementation on My Website:

When creating my visceral design code templates for texts, I will make sure the texts are using rem em instead of px, so that the texts can be resized properly when user zoomed in. I will also make sure that components I created are responsive and still appropiately shown even when zoomed in.

4. Keyboard Navigations (WCAG 2.1.1 - Level A)

User Need Identified:

From Client Brief (Value 4): " Ensure ethical, inclusive, and culturally sensitive design principles are upheld." - All users should be able to access the website using keyboard only, to ensure inclusive design principles are upheld. method.

Success Criteria:

"All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints."

Implementation on My Website:

Interactive elemtengs (elements that can be clicked, typed etc) will be focusable using Tab key, and can be activated using Enter or Space key. This will make sure buttons, links, forms etc can be accessed using keyboard only.

5. Focus Visible (WCAG 2.4.7 - Level AA)

User Need Identified:

From Johnny (Limited Tech Skills): "So if it's like... easy to navigate and doesn't have heaps of confusing menus and buttons, that'd be perfect."

Success Criteria:

"Any keyboard operable user interface has a mode of operation where the keyboard focus indicator is visible."

Implementation on My Website:

Interactive elements will have focus indicators, such as borders, different colors, additional shadow etc. These interactive elements include buttons, links, form inputs etc.

6. Reading Level (WCAG 3.1.5 - Level AAA)

User Need Identified:

From Mike: "simple language is most important for me. When website use very complicated words or long sentences, I get confused and sometimes just close the website. I like when they use everyday words that I already know."

From Client Brief (Value 2): "Provide clear, accessible information and resources to users"

Success Criteria:

"When text requires reading ability more advanced than the lower secondary education level after removal of proper names and titles, supplemental content, or a version that does not require reading ability more advanced than the lower secondary education level, is available."

Implementation on My Website:

All contents created in the websites, such the menus, descriptions, tutorials etc, will use simple language that is easy to understand. In the wiki creation feature, creators will also be encouraged to use simple yet descriptive words in order to make the content easy to understand.

7. Consistent Navigation (WCAG 3.2.3 - Level AA)

User Need Identified:

From Mike: "Clear navigation is important too. I don't like when there are too many buttons or menus everywhere. I get lost easily! Simple menu at top with clear names like "Events", "Food", "Community" - that's perfect for me."

From Johnny: "But there's definitely some apps that do my head in. Like, anything to do with school work or "educational" stuff usually sucks. They try to cram way too much information on one screen, and there's always like fifty different menus and tabs and things. I just want to find what I need quickly, not spend ages clicking through stuff."

Success Criteria:

"Navigational mechanisms that are repeated on multiple web pages within a set of web pages occur in the same relative order each time they are repeated, unless a change is initiated by the user."

Implementation on My Website:

I will use it on my website because the website I create will contain multiple pages and each of them have different content. Therefore, the consistent navigation UI and mechanism will make the website easier to be navigated by the user. In order to do this, I will create one navigation bar, and use it for every page I created.

8. Error Identification (WCAG 3.3.1 - Level A)

User Needs Identified:

  • "Event creation platform for community organized events"
  • "Timeline based system to share traditional cuisines, ideas, and events"
    • This timeline systems and event creations platform will definitely need forms to submit the data. Therefore, error identification is important to make sure the user can submit the form properly.

    Success Criteria:

    "If an input error is automatically detected, the item that is in error is identified and the error is described to the user in text."

    Implementation on My Website:

    Form validations will be implemeneted everytime there is a form in the website, such as event creation form, recipe submission form, community creation form, etc. In doing so, users will know if there is something wrong with what the form filled.

    9. Labels and Instructions (WCAG 3.3.2 - Level A)

    User Need Identified:

    From Mike (Limited English): "I also try to use Google, but sometimes I don't know what words to search. Like, should I search "multicultural community Australia" or "ethnic groups" or something else? My English vocabulary is not so big yet. My friend from India told me about something called "Meetup" - I think it's website or app? But I haven't try yet because I'm bit shy to join group where I don't know anyone.."

    From Client Brief (Value 2): "Provide clear, accessible information and resources to users."

    Success Criteria:

    "Labels or instructions are provided when content requires user input."

    Implementation on My Website:

    Forms created in the website will also have labeling and instructions on what to fill. For example, when creating recipes in the recipe submission form, there will be labels and instructions such as "Recipe Name (e.g., Rendang, Pavlova)".
